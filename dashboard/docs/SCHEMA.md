# RL Dashboard v1 Schema

This document outlines the directory structure and JSON schema generated by the RL logger API (`rl-dashboard-v1`). The API outputs structured JSON and binary files intended to be ingested by a frontend dashboard.

## Directory Structure

A single training run generates the following directory structure:

```text
runs/<run_id>/
├── run.json                  # Core metadata (env, algorithm, capabilities)
├── train_metrics.jsonl       # Append-only time-series metrics (loss, learning rate, sps)
├── episodes.jsonl            # Append-only episode statistics (return, length)
├── checkpoints.jsonl         # Append-only log of evaluation checkpoints
└── checkpoints/              # Artifacts for specific evaluation steps
    └── <step_padded>/        # e.g., 000010000/
        ├── checkpoint.json   # Checkpoint aggregate statistics
        └── rollouts/         # Selected episodes (best, median, worst)
            ├── best/         # Contains meta.json, video.mp4, signals.npz
            ├── median/
            └── worst/
```

## Core Files

### `run.json`
Metadata about the experiment.
- **`run_id`**: Unique identifier for the run.
- **`hparams`**: Hyperparameters used for training.
- **`capabilities`**: Features enabled for this run (e.g., `signals` lists the available time-series data keys).

### `train_metrics.jsonl`
Each line is a JSON object representing a training event.
```json
{"type": "train_metrics", "time": "2026-05-28T12:00:00Z", "step": 1000, "metrics": {"loss": 0.5, "sps": 400}}
```

### `episodes.jsonl`
Each line is a JSON object representing a completed training episode.
```json
{"type": "episode_end", "time": "2026-05-28T12:05:00Z", "step": 2500, "episode_return": 150.5, "episode_length": 300}
```

### `checkpoints.jsonl`
Each line points to a new checkpoint directory.
```json
{"type": "checkpoint", "time": "2026-05-28T12:10:00Z", "step": 10000, "best_return": 200.0, "checkpoint_dir": "checkpoints/000010000"}
```

## Checkpoints and Rollouts

At evaluation intervals, the logger evaluates multiple episodes. To save space, it saves only the `best`, `median`, and `worst` performing episodes.

Inside `checkpoints/<step>/rollouts/<kind>/`:
- **`meta.json`**: Rollout metadata including video availability and signal shapes.
- **`video.mp4`**: (Optional) Rendered video of the episode.
- **`signals.npz`**: (Optional) Binary NumPy archive containing time-series arrays for the episode (e.g., step-by-step rewards). Loaded into the browser as TypedArrays.

## Signal Semantics

The `signal_semantics` dictionary provides instructions to the UI on how to render `.npz` signals. While the current implementation defaults to empty, UI engineers should expect metadata mapping raw arrays to visualization preferences (e.g., units, min/max bounds, categorical labels).
